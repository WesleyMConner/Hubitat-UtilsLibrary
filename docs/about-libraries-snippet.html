This package contains a Hubitat Library.

  • A Hubitat Library is just a file of source code fragments.
  • Libraries are fantastic for isolating code that is reused by multiple Apps & Devices.
  • The source code for Apps & Devices uses "#include myNamespace.myLibrary" to copy in the content of the library (the library file).
  • As Apps & Devices are constructed they hold library content as of their creation time.
  • If the content of a library changes, the changes are seen as new Apps & Devices are constructed. The content changes are not seen by previously consctructed Apps & Devices.
Library Version Control
  • Hubitat DOES NOT provide a facility for versioning libraries.
  • HPM can deploy a library as a component of a Hubitat Bundle.
  • Bundling a library (a file) DOES NOT alter the absence of library versioning inherent to Hubitat.
  • The only real way to achieve a versioned library in Hubitat (and thus HPM) is to alter the libraries filename - e.g., myLibraryV1.groovy, myLibraryV2.groovy.
  • HPM can deliver multiple versions of the product namedmyLibrary.
    • V1 of myLibrary would be myLibraryV1_bundle with a copy of myLibraryV1.groovy.
    • V2 of myLibrary would be myLibraryV2_bundle with a copy of myLibraryV2.groovy.
    • HPM will want to update myLibrary V1 to V2.
    • Clients may elect to field some myLibrary V1 and V2 Apps & Devices which is possible since myLibraryV1.groovy and myLibraryV2.groovy can be deployed concurrently.
  • The only safe way to deploy multiple versions of a library concurrently is to use a different filename per library version.
    • #include myNamespace.myLibraryV1
      • vs.
    • #include myNamespace.myLibraryV2
